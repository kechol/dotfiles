# /team — Agent Team Orchestration Skill

## 前提条件

このスキルはClaude Code Agent Teams機能を使用する。以下が必要:

- Claude Code v2.1.32以降
- `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS` が有効化されていること

有効化されていない場合は、ユーザーに以下を案内する:

```json
// ~/.claude/settings.json
{
  "env": {
    "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1"
  }
}
```

---

## 実行フロー

`/team` を実行したら、以下の4フェーズで進行する。
**ユーザーの承認なしにチームを生成してはならない。必ずPhase 2のプラン提示後に確認を取ること。**

### Phase 1: タスク分析

ユーザーの `$ARGUMENTS` を分析し、以下を判断する:

1. **タスクの性質分類** — 以下のどれに該当するか:
   - `research`: 調査・分析・レビュー（Agent間の議論・相互批評が価値を生む）
   - `build`: 並列実装・開発（ファイル競合を避けた分担が重要）
   - `debug`: 競合仮説による原因調査（各Agentが異なる仮説を検証し互いに反証を試みる）
   - `review`: 多角的レビュー（セキュリティ、パフォーマンス、テストカバレッジ等）
   - `hybrid`: 上記の複合

2. **適切なチームサイズ** — 3〜5名を基本とする。根拠:
   - トークンコストはチームメイト数に比例して増加する
   - 6名以上は調整オーバーヘッドが利益を上回ることが多い
   - 各チームメイトに5〜6タスクが行き渡る粒度が理想

3. **Agent Teamが適切かどうかの判断** — 以下の場合はAgent Teamではなくsubagentまたは単一セッションを推奨する:
   - タスクが順次的で並列化の余地がない
   - 同一ファイルへの編集が集中する
   - タスクが小さく調整オーバーヘッドに見合わない

### Phase 2: チーム編成プランの提示

以下のフォーマットでプランを提示し、ユーザーの承認を得る:

```
## 🏗️ Agent Team Plan

**タスク**: <タスクの要約>
**分類**: <research | build | debug | review | hybrid>
**チームサイズ**: <N名>

### チームメイト構成

| # | 名前 | 役割 | 担当範囲 | モデル |
|---|------|------|----------|--------|
| 1 | <name> | <role> | <scope> | <model> |
| 2 | <name> | <role> | <scope> | <model> |
| ... | ... | ... | ... | ... |

### 協調戦略
- <Agent間のコミュニケーション方針>
- <タスクの依存関係と順序>
- <ファイル競合回避の方針（buildの場合）>
- <成果統合の方法>

このプランでチームを開始しますか？
```

### Phase 3: チーム生成と実行

承認後、Agent Teamを生成する。以下のガイドラインに従う:

#### 全タスク共通

- 各チームメイトには十分なコンテキストを含むspawnプロンプトを与える
  - 対象ファイル・ディレクトリのパス
  - 期待するアウトプットの形式
  - 他のチームメイトとの関係性・コミュニケーション指針
- タスクリストを作成し、依存関係を明示する
- チームメイトが完了するまで待機し、自分で実装を始めない

#### リードのファシリテーション責務

リードは単なるタスク分配者ではなく、議論の**ファシリテーター**として振る舞う。
以下の指針に従い、チームの思考品質と議論の生産性を能動的に管理する。

**議論の構造化:**

- チームメイトから成果・発見が報告されたら、そのまま受け取るのではなく
  **論点を構造化して整理**し、「現時点での合意点」「対立点」「未探索の領域」を
  明示した中間サマリーをチーム全体にbroadcastする
- 対立が生じた場合、双方の主張を並べて「何が事実の相違で、何が価値判断の相違か」を
  切り分けてからチームに提示する

**発言バランスの管理:**

- 特定のチームメイトばかりが発言・報告している場合、沈黙しているチームメイトに
  明示的に意見を求める（例: 「@wise-fool ここまでの議論で気になる前提はあるか？」）
- 思考ロール（critique, devils-advocate, wise-fool）には、ドメイン専門家の
  一次成果が出揃った**後に**発言を促す。先に批判が出ると議論が萎縮する

**収束の促進:**

- 議論が拡散し続ける場合、タイムボックスを設ける。具体的には
  「あと1ラウンドの応答で結論を出してください」とチームに通知する
- 合意に至らない論点は、無理に収束させず**未解決として明示的に記録**し、
  Phase 4の統合レポートに含める
- 「決定しないことのコスト」が大きい場合は、devils-advocateの手法を借りて
  チームに決断を迫る

**品質チェックポイント:**

- 全チームメイトの一次成果が出揃った時点で、以下を確認する:
  1. 当初のタスク目的に対して回答になっているか
  2. チームメイト間で矛盾する主張がないか（あれば対質を促す）
  3. 明らかに見落とされている観点がないか
- 品質が不十分な場合、追加ラウンドを設定するか、特定のチームメイトに深掘りを指示する

#### タスク分類別の協調パターン

**`research` の場合:**

- 各チームメイトに異なる調査軸・視点を割り当てる
- チームメイト同士が発見事項を共有し、互いの発見に挑戦・補完するよう指示する
- リードは各チームメイトの発見を統合し、矛盾点を解消する

**`build` の場合:**

- 各チームメイトが担当するファイル/モジュールを明確に分離する
- 共有インターフェース（型定義、API契約）を先に合意させる
- 依存関係のあるタスクは適切に順序付ける

**`debug` の場合:**

- 各チームメイトに異なる仮説を割り当てる
- 「科学的討論」モードで互いの仮説を反証するよう指示する
- 最も説得力のある仮説にチームが収束するまで議論を続ける

**`review` の場合:**

- 各チームメイトに異なるレビュー観点を割り当てる（例: セキュリティ、パフォーマンス、可読性）
- 重複を避けるため、担当領域を明確に切り分ける
- 発見事項に重要度（Critical / Warning / Info）を付与させる

#### デフォルト思考ロールの活用

以下のsubagent定義が利用可能な場合、タスクの性質に応じて自動的にチームに組み込む。
これらは特定ドメインの専門家ではなく、**チームの思考品質を高めるメタ認知ロール**である。

| Agent             | 役割                                                 | 最適なタスク分類     |
| ----------------- | ---------------------------------------------------- | -------------------- |
| `critique`        | 結論の批判的検証。論理の飛躍、証拠不足、リスクを指摘 | `review`, `research` |
| `devils-advocate` | 意図的な反論。集団思考の防止、難しい決定の促進       | `research`, `hybrid` |
| `wise-fool`       | 素朴な質問で前提を揺さぶる。暗黙の仮定の可視化       | 全タスク分類         |

**組み込みルール:**

- `research` / `review` タスク: ドメイン専門家に加えて、上記から1〜2名を自動提案する
- `debug` タスク: 各仮説検証者に加えて `wise-fool` を含めると前提の見直しに効果的
- `build` タスク: 実装完了後のレビューフェーズで `critique` を投入する構成を提案する
- ユーザーが明示的に「批判的に」「前提を疑って」「反論も含めて」等と指示した場合は、対応するロールを優先的に含める

**重要: ドメイン専門家が先、思考ロールは補助。**
チームの主体はタスク固有のドメイン専門家（セキュリティレビュアー、フロントエンド開発者等）。
思考ロールは議論の質を高める触媒として配置する。思考ロールだけのチームは避ける。

#### カスタムSubagent定義の活用

プロジェクトまたはユーザーの `agents/` ディレクトリにsubagent定義がある場合、
それらをチームメイトのロールとして活用する。例:

```
security-reviewer agentタイプを使用して、認証モジュールを監査するチームメイトを生成してください。
```

カスタムsubagentとデフォルト思考ロールは自由に組み合わせられる。例えば
`security-reviewer` + `critique` の組み合わせで、セキュリティレビューの
見落としを批判的に検証する構成が可能。

### Phase 4: 成果統合

全チームメイトの作業完了後:

1. **結果の収集** — 各チームメイトからの成果・発見事項を収集する
2. **統合レポート** — 以下の構造で統合する:
   - 各チームメイトの主要な発見・成果のサマリー
   - チームメイト間で合意した事項
   - 未解決の論点・相違点（ファシリテーション中に収束しなかったものを含む）
   - 議論の過程で覆された当初の仮定（あれば）
   - 推奨される次のアクション
3. **クリーンアップ** — チームを適切にシャットダウンする（必ずリードから実行）

## 使用例

```
# 多角的リサーチ
/team SSPMプロダクトの競合環境を調査する。技術的差別化、市場ポジショニング、価格戦略の3軸で分析

# 並列開発
/team 認証モジュールをリファクタリングする。フロントエンド、バックエンドAPI、テストの3チームで分担

# 競合仮説デバッグ
/team WebSocket接続が1メッセージ後に切断される問題を調査。サーバー側タイムアウト、クライアント側の再接続ロジック、プロキシ設定の3仮説で検証

# 多角的レビュー
/team PR #142をレビュー。セキュリティ、パフォーマンス、テストカバレッジの3観点で

# 思考ロール付きリサーチ（critique + wise-fool が自動組込み）
/team 新規プロダクトの技術スタック選定を議論する。前提を疑いながら、批判的に検討してほしい

# 意思決定の質を高める議論
/team マイクロサービス分割 vs モノリス維持の方針を決定する。反論も含めて徹底的に議論

# 引数なし（対話的に決定）
/team
```

## 注意事項

- Agent Teamはsubagentと比較して3〜7倍のトークンを消費する。コスト意識を持つこと
- 同一ファイルを複数のチームメイトが編集するとコンフリクトが発生する。`build`タイプでは必ずファイルオーナーシップを明確にする
- チームメイトがエラーで停止した場合、直接メッセージを送るか代替のチームメイトを生成する
- セッションの再開（`/resume`）はin-processモードのチームメイトを復元しない
- リードが先に完了してしまう場合は「チームメイトの完了を待ってください」と指示する
