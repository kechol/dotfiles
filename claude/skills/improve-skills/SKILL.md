# /improve-skills — Skill & SubAgent 最適化

インストール済みの Skill と SubAgent をベストプラクティスに基づいて監査し、改善提案を行う。
詳細なベストプラクティスは [references/best-practices.md](references/best-practices.md) を参照。

## ユーザー起動

`/improve-skills` と入力して実行。

## 引数

- `/improve-skills` — 全 Skill・Agent を監査して改善レポートを出力
- `/improve-skills [対象名]` — 指定した Skill または Agent のみ監査
- `/improve-skills --apply` — 改善提案を自動適用（ユーザー確認後）

---

## Phase 1: 対象ファイルの収集

### Step 1: Skill ファイルの収集

```bash
# プロジェクトレベルの Skill
find .claude/skills -name "SKILL.md" -type f 2>/dev/null

# ユーザーレベルの Skill
find ~/.claude/skills -name "SKILL.md" -type f 2>/dev/null
```

### Step 2: Agent ファイルの収集

```bash
# プロジェクトレベルの Agent
find .claude/agents -name "*.md" -type f 2>/dev/null

# ユーザーレベルの Agent
find ~/.claude/agents -name "*.md" -type f 2>/dev/null
```

引数で対象が指定されている場合はフィルタする。

## Phase 2: 各ファイルの監査

収集した各ファイルに対して、以下のチェックリストで評価する。
ベストプラクティスの詳細は [references/best-practices.md](references/best-practices.md) を参照。

### Skill 監査チェックリスト

各 SKILL.md に対して以下を評価し、0-10 でスコアリングする。

#### 1. description の品質（配点: 30%）

- [ ] 三人称で記述されているか（「〜する」「〜を行う」）
- [ ] スキルが**何をするか**が明確か
- [ ] **いつ使うべきか**のトリガー条件が含まれているか
- [ ] 使うべきでないケースが含まれているか（偽陽性防止）
- [ ] 具体的なキーワードやフレーズが含まれているか
- [ ] 1024文字以内か
- [ ] 曖昧な表現（「ドキュメントを処理」等）がないか

**スコアリング基準:**
- 9-10: 三人称 + 具体的機能 + 正負のトリガー + キーワード網羅
- 7-8: 三人称 + 機能明確 + トリガーあり（負のトリガーなし）
- 4-6: 機能は分かるがトリガー条件が不足
- 1-3: 曖昧、主語不一致、トリガーなし

#### 2. 構造と行数（配点: 20%）

- [ ] SKILL.md 本文が500行以下か
- [ ] 参照ファイルが1階層のみか（ネストなし）
- [ ] 参照ファイルへのポインタが明確か
- [ ] 100行超の参照ファイルに目次があるか

#### 3. 指示の品質（配点: 25%）

- [ ] Claude が既に知っていることの冗長な説明がないか
- [ ] 命令形（「〜する」「〜を実行する」）で書かれているか
- [ ] 一貫した用語が使われているか
- [ ] 自由度のレベルがタスクの性質に合っているか
- [ ] ワークフローが番号付きステップで構成されているか
- [ ] フィードバックループ（実行→検証→修正）があるか

#### 4. 出力フォーマット（配点: 15%）

- [ ] 期待する出力形式が定義されているか
- [ ] テンプレートや具体例が含まれているか

#### 5. 保守性（配点: 10%）

- [ ] 時間依存の情報がないか
- [ ] Windows パス（バックスラッシュ）がないか
- [ ] 選択肢の提示が過多でないか（デフォルトが提供されているか）

### Agent 監査チェックリスト

各 Agent 定義ファイルに対して以下を評価する。

#### 1. description の品質（配点: 30%）

- [ ] アクション指向の記述か
- [ ] いつ委譲すべきかが明確か
- [ ] 偽陽性を防ぐ否定条件があるか

#### 2. ツールスコーピング（配点: 25%）

- [ ] `tools` フィールドが明示的に指定されているか（省略＝全ツール継承に注意）
- [ ] ロールに必要最小限のツールのみ許可されているか
- [ ] 読み取り専用エージェントに Write/Edit が含まれていないか

#### 3. モデル選択（配点: 15%）

- [ ] `model` が指定されているか
- [ ] タスクの複雑さに対して適切なモデルか
  - 検索・単純処理 → haiku
  - 分析・レビュー → sonnet
  - 戦略・複雑判断 → opus

#### 4. プロンプトの品質（配点: 30%）

- [ ] 単一責任（1エージェント = 1ゴール）か
- [ ] 完了条件が明確か
- [ ] プロンプトが簡潔か（2000語以下推奨）
- [ ] 「なぜそうすべきか」の理由が説明されているか
- [ ] ヒューマン・イン・ザ・ループの停止条件があるか

## Phase 3: レポート生成

全ファイルの監査結果を統合レポートとして出力する。

### レポートフォーマット

```
## Skill & Agent 最適化レポート

### サマリー
- 監査対象: N Skill, M Agent
- 平均スコア: X.X / 10
- 要改善: N 件（重大: X, 中程度: Y, 軽微: Z）

### スコアボード
| 種別 | 名前 | スコア | 最大の改善ポイント |
|------|------|--------|-------------------|

### 詳細レポート

#### [Skill/Agent名]
- スコア: X.X / 10
- 発見事項:
  1. [重大度] 内容 → 推奨アクション
  2. ...
- 改善後の description 案:
  ```yaml
  description: |
    [改善案]
  ```
```

## Phase 4: 改善の適用

監査で発見した改善点を Confidence Score に基づいて自動適用する。

### Confidence Score の定義

各改善提案に Confidence Score（確信度）を付与する:

- **HIGH（9-10）**: 客観的に正しい修正。即座に自動適用する
  - description が1人称で書かれている → 三人称に修正
  - tools フィールドが未指定で全ツール継承 → ロールに応じた最小限に制限
  - description が1024文字を超過 → 短縮
  - Windows パス（バックスラッシュ）→ フォワードスラッシュに修正
  - model が未指定 → タスク複雑度に応じて設定
- **MEDIUM（6-8）**: 改善効果が高いが判断が伴う。レポートに含め、ユーザーに提示する
  - description のトリガーキーワード追加
  - 否定条件（使うべきでないケース）の追加
  - プロンプトの簡潔化
- **LOW（1-5）**: 好みやスタイルの問題。レポートに記載するのみ
  - 構造の大幅な再編成
  - プロンプトの書き換え

### 適用ルール

1. **HIGH Confidence**: 自動適用する。before/after の diff をレポートに記録する
2. **MEDIUM Confidence**: レポートに改善案を記載する。ユーザーが希望すれば適用する
3. **LOW Confidence**: レポートに提案として記載するのみ
4. **プラグイン Skill**: 変更不可。レポートに記載するのみ（プラグイン提供者への提案として）

### 適用フロー

1. HIGH Confidence の修正を即座に適用し、diff をレポートに記録
2. MEDIUM Confidence の改善案をレポートに記載
3. 適用結果を `output/improve-skills-{YYYYMMDD-HHMMSS}.md` に保存

---

## 重要ルール

- **HIGH Confidence は自動適用**: 客観的に正しい修正は確認なしで適用する
- **MEDIUM/LOW は提案のみ**: 判断が伴う修正はレポートに記載してユーザー判断に委ねる
- **プラグイン Skill は対象外**: プラグインが提供する Skill は監査対象に含めない（変更不可のため）
- **出典を明記**: ベストプラクティスの根拠を示す際は情報源を明記する
- **diff を常に記録**: 自動適用した修正は全て before/after の diff をレポートに残す
