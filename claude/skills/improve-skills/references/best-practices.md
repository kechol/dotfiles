## Skill ベストプラクティス

### SKILL.md 構造

```
skill-name/
├── SKILL.md              # メタデータ + 指示（500行以下推奨）
├── scripts/              # 実行可能コード
├── references/           # 補足コンテキスト（1階層のみ）
└── assets/               # テンプレート・静的ファイル
```

**重要な制約:**

- `name`: 最大64文字、小文字・数字・ハイフンのみ
- `description`: 最大1024文字、空でないこと
- SKILL.md 本文は500行以下を推奨（超える場合は分割）
- 参照ファイルは SKILL.md から1階層のみ（ネストしない）

### description の書き方

**必須要素:**

1. スキルが**何をするか**を三人称で記述
2. **いつ使うべきか**のトリガー条件
3. **使うべきでないケース**（偽陽性防止）

**良い例:**

```yaml
description: |
  PDF ファイルからテキスト・表を抽出し、フォーム記入・文書結合を行う。
  PDF ファイル操作、フォーム記入、文書抽出が必要な場合に使用する。
  Word 文書やスプレッドシートの操作には使用しない。
```

**悪い例:**

```yaml
description: ドキュメントの処理を手伝います
```

**三人称で書く**（description はシステムプロンプトに注入されるため）:

- 良い: "Excel ファイルを分析してレポートを生成する"
- 悪い: "私がExcelファイルを分析します"
- 悪い: "あなたはこのスキルを使えます"

**トリガーを積極的に記述する**: Claude はスキルを「使わない」方向にバイアスがかかりやすい。description にはやや積極的にトリガー条件を書く。

### プログレッシブ・ディスクロージャー

3段階のローディングシステム:

1. **メタデータ**（name + description）: 常にコンテキストに存在（約100語）
2. **SKILL.md 本文**: スキルがトリガーされたときにロード
3. **参照ファイル**: 必要に応じてロード（コンテキストペナルティなし）

**パターン1: ハイレベルガイド + 参照**

```markdown
# PDF 処理

## クイックスタート

[基本的な使い方]

## 高度な機能

**フォーム記入**: [FORMS.md](FORMS.md) を参照
**API リファレンス**: [REFERENCE.md](REFERENCE.md) を参照
```

**パターン2: ドメイン別整理**

```
bigquery-skill/
├── SKILL.md（概要とナビゲーション）
└── reference/
    ├── finance.md
    ├── sales.md
    └── product.md
```

### 指示の書き方

**自由度のレベルを課題の性質に合わせる:**

- **高自由度**（テキスト指示）: 複数アプローチが有効な場合
- **中自由度**（疑似コード・パラメータ付きスクリプト）: 好ましいパターンがある場合
- **低自由度**（具体的なスクリプト）: 操作が脆弱でエラーしやすい場合

**Claude はすでに賢い**: Claude が知っていることの説明は省く。追加すべきは Claude が持っていない文脈のみ。

**一貫した用語を使う**: 同じ概念に異なる用語を混在させない。

### ワークフローとフィードバックループ

**チェックリストパターン:**

```markdown
## ワークフロー

タスク進捗:

- [ ] Step 1: フォーム分析
- [ ] Step 2: フィールドマッピング作成
- [ ] Step 3: マッピング検証
- [ ] Step 4: フォーム記入
- [ ] Step 5: 出力検証
```

**バリデーションループ:** 実行 → 検証 → エラー修正 → 再検証

### アンチパターン

| アンチパターン             | 問題               | 解決策                     |
| -------------------------- | ------------------ | -------------------------- |
| 曖昧な description         | ルーティング不正   | 正・負のトリガーを含める   |
| SKILL.md が500行超         | トークン消費過多   | サブディレクトリに分割     |
| ネストしたサブディレクトリ | ナビゲーション困難 | ファイルは1階層のみ        |
| 散文的な指示               | 手順の抜け         | 番号付きのステップにする   |
| インラインの大きな設定     | コンテキスト浪費   | assets/ にテンプレート配置 |
| 汎用的なスクリプトエラー   | 自己修正不能       | 具体的なエラーメッセージ   |
| 時間依存の情報             | 陳腐化             | 「旧パターン」セクションに |
| 選択肢の提示過多           | 混乱               | デフォルトを提供           |
| Windows パス               | 互換性問題         | 常にフォワードスラッシュ   |

### テストと評価

**評価駆動開発:**

1. スキルなしで Claude にタスクを実行させ、失敗箇所を特定
2. 3つのテストシナリオを作成
3. ベースライン（スキルなし）を計測
4. 最小限の指示を作成しテスト
5. 反復改善

**2種類のスキル:**

- **能力向上スキル**: ベースモデルができないことを補う（モデル進化で不要になる可能性あり）
- **プリファレンスエンコーディングスキル**: チーム固有のワークフローを記述（安定的に価値あり）

## SubAgent ベストプラクティス

### 定義ファイル構造

```markdown
---
name: code-reviewer
description: コード品質とベストプラクティスのレビュー
tools: Read, Glob, Grep
model: sonnet
---

あなたはコードレビュアーです。コードの品質、セキュリティ、
ベストプラクティスへの準拠を分析してください。
```

**配置場所（優先度順）:**

1. `--agents` CLI フラグ（セッション限定、最高優先度）
2. `.claude/agents/`（プロジェクトレベル）
3. `~/.claude/agents/`（ユーザーレベル）
4. プラグインの `agents/`（最低優先度）

### description の書き方（Agent）

**必須要素:**

- アクション指向の記述
- いつ委譲すべきかを明示
- 自動委譲を促すなら "use proactively" を含める

**良い例:**

```yaml
description: "コード品質・セキュリティ・保守性のレビュー専門家。コード変更後に積極的に使用する。"
```

**悪い例:**

```yaml
description: "一般的なタスクを手伝います"
```

### ツールスコーピング

**原則:**

- `tools` を省略すると全ツールを継承（MCP含む）
- ロールベースのアクセス制御を適用する
- 読み取り専用エージェント: `tools: Read, Grep, Glob, Bash`
- 書き込み除外: `disallowedTools: Write, Edit`

**ロール別ガイドライン:**

- PM/アーキテクト: Read, WebSearch, ドキュメントアクセス
- 実装者: Edit, Write, Bash
- レビュアー: Read, Grep, Glob のみ

### モデル選択

| モデル    | 用途                                 |
| --------- | ------------------------------------ |
| `haiku`   | 高速・低コスト。大量処理、単純な検索 |
| `sonnet`  | バランス型。分析、コードレビュー     |
| `opus`    | 高推論力。戦略、複雑な判断           |
| `inherit` | 親会話と同じ（デフォルト）           |

### プロンプト設計

**単一責任**: 各エージェントは1つの明確なゴールを持つ

**完了条件を明示**: チェックリストで完了を定義する

**ヒューマン・イン・ザ・ループ**: 明示的な停止条件を埋め込む

- "受入基準が不完全な場合、番号付きの質問をして回答を待つ"
- "パブリックAPI変更やマイグレーションリスクがある場合は停止して承認を求める"

**簡潔に**: 2000語のプロンプトは賢いエージェントを作らない。遅延と保守コストが増えるだけ。

### アンチパターン（Agent）

| アンチパターン             | 問題                     | 解決策                            |
| -------------------------- | ------------------------ | --------------------------------- |
| 汎用的なエージェント多数   | 委譲の混乱               | 2-3個の集中エージェントから始める |
| 曖昧な指示                 | 委譲ガイダンスなし       | 具体的なトリガー条件を記述        |
| 巨大なプロンプト           | レイテンシ・保守コスト増 | 簡潔・直接的に                    |
| ツール冗長                 | 単一目的のタスク         | スラッシュコマンドの方が適切      |
| 過多のエージェント         | 保守困難                 | 必要な時だけ追加                  |
| ネストしたサブエージェント | 非対応                   | メイン会話からチェーンする        |

### SubAgent vs Skill の使い分け

| SubAgent を使う                  | Skill を使う                 |
| -------------------------------- | ---------------------------- |
| コンテキスト分離が必要な長い調査 | 単一目的のタスク             |
| 並列ワークストリーム             | 素早い反復アクション         |
| マルチステップの専門知識         | 別コンテキストウィンドウ不要 |
| 独立した検証が必要               | メイン会話内で完結           |
