# /markdownify — ドキュメント → Markdown 変換スキル

ローカルファイルや Google Workspace のドキュメントを Markdown に変換し、人が読める品質まで整形する。

単なる変換ツール呼び出しではなく、**取得 → 変換 → クリーンアップ → 検証** を1つのワークフローとして扱う。生出力は OCR 残渣やスキーマ由来のノイズで読みづらいため、クリーンアップ工程は省略しない。

## 前提ツール

| ツール | 用途 | 確認コマンド |
|--------|------|--------------|
| `docling` | ローカルの PDF / Office / 画像 → Markdown | `docling --help` |
| `gws` | Google Workspace（Sheets / Docs / Slides / Drive） | `gws --help` |

両方ともインストール済みでない場合は、最初にユーザーに確認する。

## 対応形式

### ローカルファイル（docling 利用）

| 形式 | 拡張子 | 備考 |
|------|--------|------|
| PDF | `.pdf` | OCR ノイズが多い。クリーンアップ必須 |
| PowerPoint | `.pptx` | スライド境界を見出しとして扱うと見やすい |
| Word | `.docx` | 比較的クリーン |
| Excel | `.xlsx` | シート単位で処理 |
| HTML | `.html` | script/style 残留に注意 |
| 画像 | `.png`, `.jpg` 等 | OCR のみ。誤認識に注意 |

### Google Workspace（gws 利用）

URL から種類を判定する:

| サービス | URL パターン | 取得方法 |
|----------|--------------|----------|
| Sheets | `docs.google.com/spreadsheets/d/<id>/...` | `gws sheets spreadsheets values get` |
| Docs | `docs.google.com/document/d/<id>/...` | `gws docs documents get` |
| Slides | `docs.google.com/presentation/d/<id>/...` | `gws slides presentations get` |
| Drive 上のファイル | `drive.google.com/file/d/<id>/...` | `gws drive files export` で PDF 等にエクスポート後 docling |

`<id>` 部分を URL から抽出する（クエリパラメータ `gid=...` などは無視してよい）。

## 処理フロー

### Phase 1: 入力確認と出力先決定

1. 入力がローカルパスか URL かを判別
2. 出力先の決定
   - **ローカルファイル**: 入力と同じディレクトリに同名 `.md` を作成
   - **URL**: ユーザーに保存先ディレクトリとファイル名を確認。指定が既にある場合（例: 「plan/references/ に保存して」）はそれに従う
3. 既存 `.md` が同名で存在する場合は上書きの可否を確認

### Phase 2: 取得・変換

#### 2-A. ローカルファイル（docling）

**必ず `--image-export-mode placeholder` を指定する**。デフォルト（embedded）だと base64 画像で出力が数MB に膨らみ、Claude が読み込めなくなる。

```bash
docling "<input-path>" \
  --to md \
  --image-export-mode placeholder \
  --output "<output-dir>"
```

絶対パスを使うことを推奨。`cd` はしない（Claude Code セッションの cwd が無効化されているケースがある）。

変換には数十秒〜数分かかる。`timeout: 600000`（10分）を指定しておくとよい。

変換完了後、出力ファイルのサイズと行数を確認:

```bash
ls -la "<output-dir>"
wc -l "<output-path>"
```

- 出力サイズが予想より極端に大きい（> 100KB/ページ相当）→ 画像が埋め込まれている可能性。再変換
- 行数が予想より極端に少ない → 変換失敗の可能性。docling のエラー出力を確認

#### 2-B. Google Sheets

全シート列挙 → 各シートのデータを取得 → Markdown テーブルに整形する。

```bash
# シート一覧取得
gws sheets spreadsheets get --params '{"spreadsheetId": "<id>", "fields": "properties.title,sheets.properties"}'

# 各シートのデータを取得（シート名で range 指定）
gws sheets spreadsheets values get --params '{"spreadsheetId": "<id>", "range": "<sheet-name>"}'
```

取得結果の `values` は2次元配列なので、1行目をヘッダーとして Markdown テーブルに変換する。シートが複数ある場合は `## <シート名>` セクションで分ける。

#### 2-C. Google Docs

```bash
gws docs documents get --params '{"documentId": "<id>"}'
```

返却は JSON 構造（`body.content` に段落・テーブル・リスト等）。段落の `paragraph.elements[].textRun.content` を連結しつつ、`paragraphStyle.namedStyleType` に応じて見出しレベル（`HEADING_1`→`#`, `HEADING_2`→`##` など）を付与する。

箇条書き（`bullet` オブジェクト）やテーブル（`table` オブジェクト）も対応する。複雑な場合はまず JSON 全体を取得して構造を確認してから変換する。

#### 2-D. Google Slides

```bash
gws slides presentations get --params '{"presentationId": "<id>"}'
```

返却の `slides[]` をループし、各スライドを `## スライド N: <タイトル>` として出力する。スライド内の `pageElements[]` から `shape.text.textElements[].textRun.content` を抽出して本文にする。

#### 2-E. Drive 上の任意ファイル

`gws drive files export` で PDF 等にエクスポートしてから docling に渡す、またはエクスポート形式が `text/markdown` に対応している場合は直接取得する。

### Phase 3: クリーンアップ（最重要）

どの取得経路でも、生出力には以下のようなノイズが含まれる。**読むのに支障がある情報は除去し、意味のある情報は必ず残す**。

Claude が直接読んで Write ツールで整形済み版を書き出す。原則として生出力を上書きして整形済み版を残す。

#### 3-1. 除去対象（ノイズ）

| 種類 | 例 | 対応 |
|------|---|------|
| 画像マーカー | `<!-- image -->` | すべて削除 |
| ページ番号 | 単独行の `1`, `2`, `3` など数字のみの行 | 削除 |
| 座標情報 / ID 文字列 | `3/#`, `IMS-D-114`, `=#7+Q×`, `0 =` | 削除 |
| OCR 誤認識の短い断片 | `klie lrs`, `BRUR`, `tE*3B*`, 孤立した2-3文字の英字 | 削除 |
| 機密表記・フッターの反復 | `Confidential`, `Visional Group only`, `» VISIONAL` が各ページ末尾に繰り返し | 冒頭に1回のみ記載 |
| HTML エンティティ残渣 | `&gt;&gt;`, `&amp;` | デコードまたは削除 |
| 空セル / null 行 | 行全体が空白・null のみ | 削除 |
| Sheet 由来の末尾空列 | すべての行で末尾セルが空 | 列ごと削除 |

**判断基準**: 「この行を消して意味が失われるか？」を自問する。失われないなら削除、失われるなら残す。迷ったら残す側に倒す。

#### 3-2. 構造の再編

- **見出しレベル調整**: タイトル→`#`、大セクション→`##`、小セクション→`###` と階層化
- **セクション統合**: スライド由来の資料では1スライド＝1セクションとして `## N. <スライドタイトル>` に整形
- **番号付け**: 論理的に順序のあるセクションには `## 1.`, `## 2.` を付ける
- **断片化された見出しの結合**: `## 発明者：` と次の行の氏名が分離している場合は結合
- **メタデータの先頭集約**: 発行元・日付・区分・出典を冒頭の箇条書きにまとめる
- **冗長な空行の圧縮**: 3行以上続く空行は1行に

#### 3-3. テーブル整形

生出力の表出力には以下の問題がよく出る:

- **列ラベル重複**: `| 非保証業務 | 非保証業務 |` のように同じラベルが並ぶ
- **ヘッダー行が2行に分断**: 1行目 `| 項目 | 保証業務 | 非保証業務 |`、2行目 `| 項目 | 監査 | AUP |` のように分かれる
- **セルの改行崩れ**: セル内改行が行区切りになり表が壊れる
- **Sheet 由来の長い表**: ヘッダーと本文の区別が難しい

**対応**:
- ヘッダー2行分を結合し、意味のある単一ヘッダーにする（例: `監査（保証業務）`, `AUP（非保証業務）`）
- 重複列は統合、または内容が異なるなら別の見出しを付ける
- セル内改行はスペースまたは句読点で連結して1セルに収める
- 巨大な表は意味のある単位でセクション分割することも検討

#### 3-4. 箇条書きの整形

- `- 1R`, `· d1` のような OCR 断片は削除
- 番号付きリストに整形すべき箇所（手順・メリット列挙等）は `1.`, `2.` に変換
- **太字の活用**: 項目名と説明が並ぶ場合は `- **項目名**: 説明` 形式に

### Phase 4: 検証

クリーンアップ後、以下を確認:

1. **内容の正確性**: 元データと見比べ、**意味が変わっていないか**を確認する。要約・意訳はしない。原文の表現を可能な限り保持する
2. **固有名詞**: 人名・製品名・日付・数値の書き換えは厳禁
3. **構造の論理性**: 見出し階層が自然か、セクション順序が原文と一致しているか
4. **Markdown の妥当性**: テーブルが崩れていないか、リストが正しくレンダされるか

検証中に元情報の欠落が見つかったら、生出力を再確認して補完する。

## 重要原則

1. **クリーンアップを省略しない**: 生出力をそのまま保存するのは不完全な成果物。常に Phase 3 を実施する
2. **内容の正確性 > 見た目の美しさ**: 意訳・要約・言い換えは行わない。読みやすさのための構造化と、明確なノイズ除去に留める
3. **迷ったら残す**: 情報の要否で判断に迷う場合は削除せず残す。失われた情報は復元できない
4. **冪等性**: 同じ入力に対して同じ出力が得られるよう、再変換可能な手順にする（オプション指定を明文化）
5. **出典の明記**: 整形後 Markdown の冒頭に元ファイル名・URL と取得日を記載する
6. **外部アクセスへの配慮**: Google Workspace の取得は外部 API アクセスを伴う。機密ドキュメントの場合はユーザーに確認する

## 出力フォーマット規約

整形済み Markdown の冒頭は以下の構造を推奨:

```markdown
# <ドキュメントタイトル>

- **発行元**: <組織名>
- **日付**: <YYYY-MM-DD または元ドキュメント記載の日付>
- **区分**: <機密区分がある場合>
- **出典**: <元ファイル名 または URL>（<docling / gws> で取得・変換後、手動整形）
- **取得日**: YYYY-MM-DD

---

## 1. <最初のセクション>
...
```

末尾には元ドキュメントの著作権・機密表記がある場合はそれを再掲する:

```markdown
---

*<元ドキュメントの機密・著作権表記>*
```

## 運用のコツ

- **PPT 由来の PDF** は特にノイズが多い（図表内の断片テキストが大量に混入）。クリーンアップに時間をかける
- **複数ページの大型資料**（50ページ超）は一度 `limit` で先頭を確認し、構造を把握してから全体整形に着手する
- **表が多い資料**（財務資料・比較表等）は表の整形に最も時間がかかる。生出力の表を1つずつ目視チェックする
- **Google Sheets の巨大スプレッドシート** はシート単位で取得し、シート名を `##` セクション見出しにする。1シートが巨大なら抜粋範囲をユーザーと合意する
- **Google Docs の画像・埋め込み** は gws では取得しきれない場合がある。必要なら PDF エクスポートして docling に回す
- **機密資料**（Confidential / NDA 対象）は外部 API に送信しないこと。docling / gws はどちらも認証済みチャネルを経由するが、整形時に無関係な外部 LLM に丸投げしない

## 典型的な失敗パターン

| 失敗 | 原因 | 対策 |
|------|------|------|
| 出力が数MB | `--image-export-mode placeholder` 未指定 | 再変換 |
| `cd: no such file` | Claude Code の cwd 失効 | 絶対パスで指定 |
| タイムアウト | 大型資料での変換遅延 | `timeout: 600000` 指定 |
| 意味が変わってしまった | 過度な要約・言い換え | 原文保持の原則に戻る |
| テーブルが壊れた | セル内改行の処理ミス | 生出力を再確認して手動修正 |
| gws が認証エラー | OAuth トークン切れ | ユーザーに再認証を依頼 |
| URL から ID 抽出ミス | クエリパラメータを含めてしまった | `/d/<id>/` の `<id>` のみを取り出す |
